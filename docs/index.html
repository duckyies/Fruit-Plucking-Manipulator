Fruit Plucking Manipulator

Autonomous Harvest via YOLO Detection & Depth-Guided Control

An autonomous robotic system that uses a wrist-mounted Intel RealSense RGB-D camera, a custom-trained YOLOv8 model, and a 6-DOF Doosan A0509 cobot to detect and harvest bananas in real time — with statistically validated depth estimation and iterative approach control.

YOLOv8 Intel RealSense D435i Doosan A0509 DH Gripper ROS 2 Python
Explore the Project

Project Overview

This project builds an end-to-end autonomous fruit-harvesting pipeline. A Doosan A0509 collaborative robot arm, equipped with a wrist-mounted Intel RealSense D435i camera and a DH Robotics parallel gripper, scans its environment by sweeping Joint 5. When the onboard YOLOv8 detector spots a banana with sufficient confidence, the system enters a 2-second depth-validation window — collecting multiple depth readings, rejecting statistical outliers (σ-filtering), and averaging the survivors to obtain a stable 3D target. The robot then approaches iteratively, recalculating depth at each 50 mm step via a hand-eye calibration matrix, before closing the gripper and returning to rest.

Project Objectives

🍌

Fruit Detection

Detect ripe and unripe bananas in real time using a YOLOv8 model trained on a custom dataset, with per-class confidence thresholding.

📐

Depth Estimation

Use the RealSense depth stream to obtain robust 3D fruit positions, validated over a 2-second window using σ-based outlier rejection.

🔄

Hand-Eye Calibration

Transform detected fruit coordinates from the camera frame to the robot's base frame using a pre-calibrated T_tool_cam extrinsic matrix.

🤖

Iterative Approach

Navigate the arm toward the target in 50 mm steps with mid-approach depth recalculation to correct for noisy initial estimates.

✊

Gripper Control

Drive the DH Robotics parallel gripper via Modbus RTU over USB to open, close, and recover autonomously during the pick sequence.

🖥️

Live GUI

Provide an operator Tkinter interface showing the annotated camera feed, Cartesian position, joint angles, and manual control buttons.

Methodology

1. YOLOv8 Fruit Detection

1.1 Custom Dataset & Training

  • A labelled dataset of banana images was assembled covering ripe and unripe classes under varied lighting and pose conditions.
  • YOLOv8 (ultralytics) was fine-tuned on this dataset; the best checkpoint is stored at runs/detect/train6/weights/best.pt.
  • Both classes share a label containing "ripe", allowing the system to filter targets with a simple string match.

1.2 Real-Time Inference

  • Each camera frame is passed to model(frame, verbose=False) on every GUI tick (≈ 30 ms cadence).
  • Detections are drawn with bounding boxes: red for ripe/unripe targets, green for others.
  • A minimum confidence threshold of 0.70 is enforced before a detection triggers any robot motion.

1.3 Detection Gate

  • Detection is only active when search_active = True — set after the robot has settled at its search pose — to prevent false triggers during motion.
  • Once a target is locked (target_locked), detection is suppressed for the rest of the pick cycle.

2. Depth Estimation & Statistical Validation

2.1 Depth Deprojection

  • The RealSense SDK function rs2_deproject_pixel_to_point(intrinsics, [cx, cy], depth) converts the bounding-box centroid pixel into a metric 3D point (X, Y, Z) in the camera frame.
  • Readings where depth ≤ 0 or depth ≥ 2.0 m are discarded as out-of-range.
  • Z is sign-flipped (−Z) to align with the robot's coordinate convention.

2.2 2-Second Collection Window

  • On the first detection of a banana, the system starts a 2-second accumulation phase rather than moving immediately.
  • Every subsequent frame that still detects the banana appends a new (X, Y, Z) reading to banana_readings.
  • A live countdown overlay ("Validating depth… 1.4 s") is drawn on the camera feed.
  • If the banana disappears mid-collection, the buffer is cleared and the phase resets.

2.3 σ-Based Outlier Rejection

  • After the window closes, filter_and_average(readings, sigma=2.0) is called.
  • The mean and standard deviation are computed per axis (X, Y, Z).
  • Any reading whose value on any axis falls outside mean ± 2σ is discarded.
  • If fewer than 3 clean readings survive, the move is aborted and detection resets — preventing erroneous grasps.
  • The survivors are averaged to produce a single stable 3D target coordinate.

3. Hand-Eye Calibration & Coordinate Transform

3.1 Calibration Matrix

  • The 4 × 4 homogeneous matrix T_tool_cam encodes the rigid transform from the camera frame to the robot tool (flange) frame.
  • It is loaded once at startup from T_tool_cam.txt and printed for operator verification.
# Loaded transform (tool → camera)
T_tool_cam =
  [[ 0.890 0.455 -0.030 | -74.8 mm ]
   [ 0.455 -0.886 0.091 | 11.5 mm ]
   [ 0.015 -0.095 -0.995 | 14.9 mm ]
   [ 0.000 0.000 0.000 | 1.000 ]]

3.2 Base-Frame Transform

  • get_current_posx() returns the TCP pose as [x, y, z, rz1, ry, rz2] (ZYZ Euler, mm + degrees).
  • doosan_tcp_to_matrix(tcp) converts this into a 4 × 4 homogeneous matrix T_base_tool.
  • The fruit point in the camera frame is promoted to homogeneous coordinates and chained: P_base = T_base_tool @ T_tool_cam @ P_cam.
  • The result is converted back to mm for the Doosan API.

4. Scanning Strategy

  • The robot first moves linearly to a fixed SEARCH_POSX Cartesian pose.
  • Joint 5 (wrist tilt) then sweeps from −90° to +90° in 5° steps at 20 °/s, panning the wrist camera across the workspace.
  • Scanning runs in a dedicated daemon thread; detection is only enabled after the robot reaches the search pose to avoid motion blur false-positives.
  • When a target is validated, scanning halts (self.scanning = False) and the approach begins.

5. Iterative Approach & Mid-Step Recalculation

5.1 Step-Based Motion

  • The robot advances toward the target in 50 mm linear steps using movel().
  • If the initial distance exceeds 500 mm, a 200 mm safety margin is subtracted from the first step.
  • When the remaining distance falls within 50 + 150 mm, a final approach move stops 150 mm short of the target before a small joint-space dip (J2 − 3°) to lower the gripper onto the fruit.

5.2 Mid-Approach Depth Recalculation

  • After each 50 mm step, a fresh single-frame depth estimate (_recalculate_banana_depth) is obtained.
  • If the new base-frame distance is more than 80 mm closer than the current target, the target is updated — correcting for depth errors in the initial estimate.
  • Otherwise the original target is kept, avoiding jitter from noisy single-frame readings.

5.3 Grip Sequence

  • The gripper is pre-closed at the start of the approach for safety and collision detection.
  • On reaching the fruit, dh_gripper.open() is called, the arm pauses 3 s, then returns to REST_JOINTS.
  • (In close-range scenarios, a direct close → rest shortcut is used.)

DH Robotics Gripper

The DH Robotics parallel gripper serves as the end-effector and is controlled via Modbus RTU over a USB serial link (/dev/ttyUSB0, 115 200 baud). A custom Python driver wraps minimalmodbus with auto-recovery on slave fault.

  • Two-finger parallel gripper with high grip precision
  • Compact, lightweight — ideal for cobots
  • Position control via register 0x0103 (0 = open, 1000 = fully closed)
  • Speed (register 0x0101) and force (register 0x0102) configurable
  • Auto-homing on initialisation via register 0x0100
  • Automatic re-initialisation on SlaveReportedException

Intel RealSense D435i Camera

Overview

  • The D435i is mounted at the robot wrist (tool flange), giving it an eye-in-hand configuration that moves with the arm during scanning.
  • Streams 640 × 480 colour and depth at 30 FPS via pyrealsense2.
  • Depth and colour frames are aligned implicitly by using the same centroid pixel on both streams.

Camera Data Streams Used

  • RGB Stream — fed to YOLOv8 for fruit detection and GUI display.
  • Depth Stream — pixel-wise distance used for 3D localisation and σ-validation.

Robot Operating System (ROS 2)

Role in This Project

  • The system is initialised with rclpy.init() and runs as a ROS 2 node named banana_gui under namespace dsr01.
  • The Doosan driver is loaded via DR_init, which patches the ROS node handle into the robot SDK.
  • The camera capture loop runs in a separate Python threading.Thread, decoupled from the GUI event loop.

Node Architecture

  • Camera Thread — continuously pulls frames from the RealSense pipeline and stores them in shared memory protected by a threading.Lock().
  • GUI / Detection Thread — reads frames, runs YOLO, manages the depth-collection state machine, and updates the Tkinter display at 30 ms intervals.
  • Motion Threads — each motion command (search, approach, rest) spawns a daemon thread so the GUI remains responsive.
  • Gripper Thread — initialisation and grip cycles run asynchronously to avoid blocking.

Doosan Robot API (DSR_ROBOT2)

The DSR_ROBOT2 package exposes a Python interface to the Doosan DART controller over ROS 2. The project uses the following API functions:

Motion Commands

  • movel(posx, v, a, ref, mod) — moves the TCP along a straight Cartesian line to the target pose. Used for all approach steps and the search-pose move.
  • movej(posj, vel, acc) — moves the robot in joint space. Used for rest-pose transitions and the final wrist-dip during approach.
  • posj(j1…j6) — constructs a joint-position data object from six angles.

State Reading

  • get_current_posx() — returns the TCP's current Cartesian pose [x, y, z, rz1, ry, rz2] used to chain the hand-eye transform.
  • get_current_posj() — returns joint angles [J1…J6] used during scanning and final-approach adjustments.

Robot Mode

  • set_robot_mode(ROBOT_MODE_AUTONOMOUS) — enables full autonomous execution without teach-pendant confirmation.
  • set_velx / set_accx — global Cartesian velocity and acceleration limits applied at startup.

Key Control Functions

go_search()

Disables detection, drives the arm to SEARCH_POSX via movel(), then spawns the scan_j5() thread and re-enables detection.

scan_j5()

Sweeps Joint 5 from −90° to +90° in 5° steps, continuously panning the wrist camera to find fruit in the workspace.

update_camera()

Core 30 ms GUI loop. Runs YOLO on the latest frame, manages the depth-collection state machine, updates the annotated video display.

_start_collection()

On first banana sighting, initialises the 2-second depth reading buffer and timestamps the start of the validation window.

_finish_collection()

Calls filter_and_average() on the buffer; if enough clean readings remain, extracts a stable 3D target and triggers move_to_banana().

camera_to_base()

Chains T_base_tool @ T_tool_cam @ P_cam to transform a camera-frame 3D point into the robot base frame in mm.

move_to_banana()

Iterative approach loop: 50 mm steps with mid-step depth recalculation, final 150 mm hold-off, wrist dip, and grip sequence.

filter_and_average()

σ-based outlier rejection on a list of (X, Y, Z) readings; discards values beyond 2σ per axis and averages the survivors.

go_rest()

Halts scanning and detection, runs a gripper open/close test cycle, then drives the arm to REST_JOINTS in joint space.

Results & Discussion

🎯

Detection Performance

YOLOv8 achieved reliable detection of bananas at the 0.70 confidence threshold across varied lighting and orientations.

📊

Depth Validation

The 2-second σ-filter window successfully eliminated noisy depth spikes, providing a stable 3D target before every approach.

🔗

System Integration

All subsystems — ROS 2, camera thread, YOLO, gripper driver, and GUI — ran concurrently without deadlock or frame drops.

⚠️

Challenges

Depth accuracy degraded near the edges of the RealSense's valid range; the mid-approach recalculation loop was added to compensate.

🔮

Future Work

Multi-fruit prioritisation, stem detection for precise plucking angle, and integration with a mobile base for field deployment.
